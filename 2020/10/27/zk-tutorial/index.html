ZooKeeper基础使用

发表于 2020-10-27 分类于 ZooKeeper Valine：

ZooKeeper的应用场景

  ZooKeeper是一个经典的分布式数据一致性解决方案，致力于为分布式应用提供一个高性能、高可用，且具有严格顺序访问控制能力的分布式协调存储服务。它可以

  1. 维护配置信息（作为分布式配置中心）
  2. 分布式锁服务
  3. 集群管理
  4. 生成分布式唯一ID

关于严格顺序访问：对于来自客户端的每个更新请求，ZooKeeper都会分配一个全局唯一的递增编号，这个编号反应了所有事务操作的先后顺序。

ZooKeeper的数据模型

  ZooKeeper节点在结构上表现为树状，树中的各节点称为znode，一个znode可以有多个子节点。ZooKeeper使用路径来定位某个节点，比如/com/anyoptional/zookeeper，com、anyoptional和zookeepr分别是根节点、一级节点和二级节点，其中com是anyoptiona的父节点，anyoptional是com的子节点，依此类推。

  znode兼具文件和目录两种特点，它既像文件一样维护着数据和元信息，又像目录一样可以作为路径标识的一部分。大体上，znode可以分为三部分

  1. 数据域，即data
  2. 子节点，即child nodes
  3. 元数据，即stat，用来描述当前节点的创建和修改记录等

节点状态stat包括

  • cZxid，节点创建时的事务id
  • ctime，节点创建时的时间
  • mZxid，节点最后一次更新时的事务id
  • mtime，节点最后一次更新时的时间
  • pZxid，当前节点的子节点最后一次修改时的事务id
  • cversion，子节点的更改次数
  • dataVersion，节点数据的更改次数
  • aclVersion，节点ACL的更改次数
  • ephemeralOwner，如果节点是临时节点，表示创建该节点时的session id，如果是持久节点，恒为0
  • dataLength，数据内容的长度
  • numChildren，子节点数量

ZooKeeper中的节点分为两种，临时节点和持久节点，节点的类型在创建时确定并且不能修改。

  • 临时节点，该节点的生命周期依赖于创建它的会话，一旦会话结束，临时节点将被删除，当然也可以手动删除。虽然临时节点都会绑定到一个会话上，但它们对所有的客户端也是可见的。另外，临时节点不能拥有子节点
  • 持久节点，该节点的生命周期不依赖于会话，并且只有在客户端显示执行删除操作时，才会被移除

ZooKeeper常用命令（ZooKeeper CLI version: 3.6.2）

1. create 创建节点
2. set 更新节点
3. delete 删除节点（仅在节点为叶子节点时有效）
4. deleteall / rmr 删除节点（即是有子节点也可以删除）
5. get 查看节点
6. stat 显示状态（元数据）
7. getAllChildrenNumber 获取指定节点的子节点数量
8. getEphemerals 获取当前会话下的所有临时节点
9. ls 显示子节点列表

ZooKeeper的权限控制

  ZooKeeper类似文件系统，客户端可以创建、更新或删除节点。ZooKeeper使用ACL（access control list）来实现权限控制，包含scheme:id:permisson三个部分

  • scheme，授权模式
    • world，只有一个用户anyone，代表登录zookeepr的所有人（默认）
    • ip，对客户端使用ip地址认证
    • auth，使用已添加认证的用户认证
    • digest，使用『用户名:密码』的方式认证（密码必须是加密过的）
  • id，授权对象，权限赋予的实体，比如实际的ip地址或用户
  • permission，授予的权限，包括增、删、读、写和管理5种权限，其中delete是指对子节点的删除权限，其它4种是对节点自身的操作权限
    • c，create
    • d，delete
    • r， read
    • w，write
    • a，admin

ACL的特性如下

  • 权限控制基于znode，因此需要针对每个节点进行权限控制
  • 每个节点都支持设置多个权限和多种权限控制方案
  • 子节点不继承父节点的权限，客户端可能无权访问某个节点，但却可以访问它的子节点
1. setAcl 设置节点权限
2. getAcl 获取节点权限
3. addauth 添加一个授权用户

ZooKeeper事件监听机制

1. Watcher概念

  ZooKeeper提供了事件的发布/订阅功能，多个订阅者可同时监听某一特定主题，当该主题自身状态发生变化时（比如节点内容改变、节点下的子节点列表改变等），可以实时、主动地通知所有订阅者。

  ZooKeeper采用Watcher机制来实现事件的发布/订阅功能，该机制在被订阅的主题发生变化时会异步通知客户端，因此客户端不必在注册Watcher后轮询结果。

2. Watcher架构

  Watcher实现由三个部分组成

  1. ZooKeeper服务端
  2. ZooKeeper客户端
  3. 客户端的ZKWatchManager对象

客户端首先将Watcher注册到服务端，同时将Watcher保存到ZKWatchManager中。当ZooKeeper服务端检测到订阅的主题发生变化时，它会主动通知客户端，并由客户端的ZKWatchManager来协调相关的Watcher进行事件处理。

3. Watcher的特性
  1. Watcher是一次性的，一旦触发便会被移出，再次使用需要重新注册
  2. Watcher是串行执行的，因此回调逻辑不宜过重（只有回调后客户端才能观察到新的数据状态）
  3. Watcher只有在当前session彻底失效后才会无效，若在session有效期内快速重连成功，Watcher仍能发挥作用

ZooKeeper应用实例

1. 作为配置中心——事件监听机制的应用

  设计思路：

  1. 在ZooKeeper中建立一个根节点，比如/CONFIG，代表某个配置文件
  2. 将配置文件中的信息作为根节点的子节点存储，比如配置项timeout=3000，在Zookeeper中展现为：/CONFIG/timeout ，节点内容是3000
  3. 所有使用到该配置信息的应用集成ZooKeeper并监控/CONFIG的状态，一旦配置信息也就是子节点发生变化，每个应用就会收到ZK的通知
  4. 从ZK中获取新的配置信息应用到系统中

Srping Cloud Zookeeper Config就是基于这种思路构建的。

2. 分布式ID生成器

  基于持久有序节点：

  1. 在ZooKeeper中建立一个持久有序的节点，比如/id，ZK会返回节点名 + 递增序号作为最终节点名
  2. 取出递增序号返回，并删除前一个生成的节点，也就是序号比当前低的那个节点

  基于dataVersion：

  1. 在ZooKeeper中建立一个持久节点，比如/id
  2. 使用set命令更新节点值，返回新的dataVersion
3. 分布式锁

  设计思路

  1. 在ZooKeeper中建立一个持久节点，比如/locks，每个客户端在这个节点下都各自创建一个临时有序节点，比如/locks/lock0000x
  2. 客户端取得/locks下的所有子节点并进行排序，判断排在第一位的是不是自己创建的节点，如果是，表示获得锁成功
  3. 如果不是，则监听前一个节点，比如节点是/locks/lock0003那么就监听/locks/lock0002
  4. 当前一个节点对应的客户端完成了工作释放了锁，也就是删除了它所创建的节点，将会触发监听客户端的逻辑
  5. 监听客户端回到步骤2，再次判断自身是否获得了锁

这种思路创建的是fair lock。

创建临时节点可以保证在故障时也可以释放锁，比如客户端a获得了锁后没来得及释放就宕机了，如果是持久节点那就永远不能释放了，而如果是临时节点，在ZK通过心跳机制检测到客户端a下线，可以关闭会话释放临时节点。

在步骤2-3中获取子节点列表与设置监听这两步操作的原子性问题。考虑这样一个场景：客户端a对应子节点为/locks/lock0001，客户端b对应子节点为/locks/lock0002，客户端b获取子节点列表时发现自己不是序号最小的，但是在设置监听器前客户端a完成业务流程删除了子节点/locks/lock0001，客户端b设置的监听器岂不是丢失了这个事件从而导致永远等待了？这个问题不存在的。因为zookeeper提供的API中设置监听器的操作与读操作是原子执行的，也就是说在读子节点列表时同时设置监听器，保证不会丢失事件。

ZooKeeper监控命令

  ZooKeeper支持某些4字命令与其交互，这些命令大多是查询命令，用来获取ZooKeeper服务的当前状态及相关信息，可以通过telnet或nc向ZooKeeper提交相应命令。

1. conf——输出ZK server的详细配置信息
2. cons 列出所有连接到ZK servr的会话的详细信息
3. crst——重置所有连接到ZK server的会话信息
4. dump 列出所有未经处理的会话和临时节点
5. envi 输出服务器环境的详细信息
6. ruok 测试ZK server是否处于正确运行状态
7. stat 输出ZK server的详细信息
8. srst 重置ZK server状态
9. wchs 输出ZK server中watcher的简洁信息
10. wchc 通过会话分组，列出被watch的所有节点
11. wchp 通过节点路径分组，列出被watch的所有节点
12. mntr 输出集群的健康状态
